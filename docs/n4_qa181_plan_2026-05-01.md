# N4 QA=181 re-quantisation — feasibility & plan

**Status (2026-05-01):** feasibility checks done; implementation queued (2-3 days).
**Context:** post-T1 training queue item from `docs/training_methodology_cross_engine_2026-05-01.md`.
**Origin:** Viridithas net #80 (`qa-181`), STC +16.76 ±6.68 vs `artemis-qa-255`. Re-quant of an already-trained net using cheaper SIMD.

## Why QA=181 wins for viri (and why our gain is smaller)

The headline +16.76 STC came from two compounding effects:

1. **i8 FT (L0) weight storage instead of i16.** At QA=181, weights bound by `±0.99 × QA = ±179.2` mostly fit in i8 `[-127, 127]` after rescale. Halves L0 weight memory bandwidth on accumulator update — the dominant cost of a v5-style FT-then-output net.
2. **Cheaper SCReLU pack.** `v² / 181` requires a multiply-high (`(v² × 362) >> 16`) instead of viri's old i32 path. Mixed: marginally more expensive than the `>> 8` divide-by-255 shortcut Coda uses, but enables i8 packing downstream.

**Coda v9 ≠ viri qa-181 architecture.** v9 has hidden layers (16 → 32) and a 51 MB threat-weight block that already dominates inference cost (per Zeus's
`docs/nps_structural_findings_2026-05-01.md`). The FT block (5.9 MB at i16, 3 MB at i8) is a much smaller fraction of total weight bandwidth. So the i8-FT lever still helps but by a smaller margin.

**Expected return: +3 to +8 Elo** (vs viri's +16.76).

## Feasibility: prod net weight clip rates at i8 storage / QA=181

Tool: `scripts/qa181_feasibility.py`. Run on `net.nnue` (current prod, v9-768th16x32-kb10-w15-e800s800-crelu-C8fix-factor):

| Block | Count | Abs max @ QA=255 | Abs max @ QA=181 | i8 clip rate post-rescale | Verdict |
|---|---:|---:|---:|---:|---|
| FT (PSQ) weights | 5.9M | 252 | 178.9 | **0.064%** | ≈ borderline / feasible |
| FT biases | 768 | 381 (1.49×QA) | 270 | 1.56% | Keep i16 (no bandwidth issue at 768 values) |
| Threat weights | 51M | 127 | 90 | 0% | ✓ trivial — already i8 at QA=255 |

**FT i16→i8 at QA=181 is feasible.** Bullet's `--fld-clip 0.99` keeps virtually all FT weights in range. The 0.064% clip rate (~3.7K of 5.9M weights) is the precision floor — likely costs <1 Elo, viri net-won despite a similar floor.

**The threat block doesn't benefit.** Already i8. QA=181 doesn't fix Zeus's L3-residency
target (51 MB → won't shrink); only the 12 MB FT block goes 12 → 6 MB. They stack as orthogonal levers but neither subsumes the other.

## Implementation plan

### Phase 1: Coda inference (1.5 days)

**1a. Audit `src/nnue.rs` `255` literals.** `grep -n 255 src/nnue.rs` returns ~30 sites; classify into:
   - **Replace with `QA`** (~6 sites): tail-loop `clamp(0, 255)` at lines 703, 769, 1302, 1726, 1794, 1727 — already QA-symbolic, just replace the literal.
   - **Replace with `(v² × 362) >> 16` approximation** (~3 sites): SCReLU pack `>> 8` divide-by-255 at lines 692-693 (AVX2), 1313 (AVX-512), 1700/1710 (NEON). 362/2^16 ≈ 1/181.0 to <1% error. Verify with `(32761 × 362 + 32768) >> 16 = 181` (rounds correctly).
   - **Recalibrate `FT_SHIFT`** (1 const): currently 9 to give L1 input range `[0, 127]` from `QA² = 65025`. At QA²=32761, target same L1 range → `FT_SHIFT = 8`. Check `PW_SCALE = (QA*QA) >> FT_SHIFT` invariant downstream.

**1b. Add i8 FT path.** New accumulator-update kernels for i8 FT weights (AVX2/AVX-512/NEON). Existing i16 path stays for back-compat with QA=255 nets. Gated on a header flag (new `int8_ft` bit in `flags` byte).

**1c. Tests.** Bench-equivalence smoke test: load prod-QA255 net, evaluate; load same net re-quantised to QA=181; eval should match within ~1 cp absolute on most positions. Existing `cargo test` suite must pass.

### Phase 2: Bullet f32 → QA=181 re-quantisation (0.5 day)

**Why from-checkpoint, not double-rounded.** Re-quantising the existing i16 .nnue (`weight_181 = round(weight_255 × 181/255)`) introduces double-rounding error on top of the precision change. Viri re-quantised from f32 checkpoint to get +16.76 cleanly. Going through the i16 path adds ~0.5 cp/dim noise — fine for math correctness verification, not fine for the SPRT.

**Approach.** Modify `adamtwiss/bullet` fork to expose `--qa 181` and `--int8-ft` save flags. Re-run the quantise step on the existing prod checkpoint (no retraining). Output a new `.nnue` at QA=181 with i8 FT storage.

**GPU needed:** loading the f32 checkpoint + quantising is fast (~minutes). One free GPU on any host for one short run.

### Phase 3: SPRT + retune (~6h fleet)

**3a. SPRT [-3, 3] vs main** with the QA=181 net + QA=181 inference path. Tests: does the precision change net-win or net-lose at fixed search thresholds?

**3b. EVAL_SCALE recalibration if needed.** `coda eval` RMS over 500 sampled positions; compare to prod ~580. If shifted, set EVAL_SCALE = 400 × prod_rms / new_rms. Per CLAUDE.md, preferred fix is a focused SPSA retune, not EVAL_SCALE adjustment.

**3c. Focused retune** of search thresholds (RFP, futility, SEE, LMR, history magnitudes — ~15 params) if EVAL_SCALE shifted by >5%. ~1500 iter on the QA=181 net. Then SPRT [0, 3] retuned-QA=181 vs main.

## Open risks

| Risk | Mitigation |
|---|---|
| FT_SHIFT=8 changes L1 input dynamic range; trained L1 weights expect a specific scale | Bench-equivalence test catches; if eval shifts >5cp, recalibrate via L1 weight rescale or focused retune |
| Activations during bench may exceed 181 frequently → SCReLU clamp loses signal | Pending instrument-and-measure step (~1h, deferred) |
| `>>16` mul-high adds 1 cycle/iter to SCReLU pack vs `>>8` shift | Acceptable; SCReLU pack is small fraction of total inference |
| Bullet f32 checkpoint may be archived only as .nnue (lossy) | Verify checkpoint exists for prod recipe before scoping |

## Decision point

QA=181 is **feasible and queued** but not the highest-ROI training-side experiment in the queue. ROI ranking from `docs/training_methodology_cross_engine_2026-05-01.md`:

1. N3 Arasan recipe stack (highest precedent, 4-line config edit, +5-15 Elo)
2. **N4 QA=181** (this doc, +3-8 Elo, 2-3 days work)
3. N5 sibilant L1-on-activations (+5.9 LTC viri precedent)
4. N1 eleison WDL ramp + EVAL_SCALE

QA=181 is firmly mid-priority. Worth doing once T1 pipeline-stability gate clears and either N3 has resolved or we want orthogonal work in parallel.
