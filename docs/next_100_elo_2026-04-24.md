## Six levers, ranked by expected contribution

### 1. Shelved-net deployment (~30 Elo, already SPRT-proven, cheapest)

Two already-validated architectural choices that haven't been
trained as full SB800 production replacements:

- **Reckless-KB layout** (+15 Elo SPRT) — 10 king buckets in
  Reckless's specific geometry, vs current consensus-KB.
- **Factor architecture** (+15 Elo SPRT) — shared PSQ weights across
  king buckets via l0f layer, folded at save time.

Both proven at shorter TC (~SB400) with independent SPRT H1s. Never
run as a combined SB800 full-strength net. Single training run gets
both, ~40 hours GPU on RTX 4070S.

**This is the single cheapest 30 Elo we'll ever get.** Don't let it
sit shelved while we chase 5 Elo search wins.

### 2. Reckless output bucket layout (~5-15 Elo, untested, bundle with #1)

**New idea** surfaced 2026-04-24. Reckless's output-bucket layout
maps piece count to bucket differently from ours:

| popcount | Coda bucket | Reckless bucket |
|---|:---:|:---:|
| 2-8 | 0-1 (two buckets) | **0 (all nine piece counts in one bucket)** |
| 9-12 | 1-2 | 1 |
| 13-17 | 3 | 2 |
| 18-21 | 4 | 3 |
| 22-25 | 5 | 4-5 |
| 26-29 | 6 | 5-6 |
| 30-32 | 7 | 7 |

Coda uses uniform `(piece_count - 2) / 4`. Reckless concentrates
granularity in middlegame (3 piece counts per bucket) and pools the
entire endgame (2-8 pieces) into ONE bucket.

**Two compounding wins predicted:**

1. **Data density in the rare-position regime**: 2-8 piece positions
   are underrepresented in T80 data. Splitting across buckets means
   each bucket gets noisy per-weight training. Pooling them gives
   bucket 0 roughly 2× the data density → smoother convergence.
2. **Smoother transitions in the endgame phase**: Coda currently has
   bucket boundaries inside the endgame (pieces 2-8), so trading a
   piece in a K+P vs K endgame can jump buckets and create eval
   discontinuities. Reckless has zero boundaries inside that range.

**Cost:** training-time architecture change. One lookup table edit
in `examples/coda_v9_768_threats.rs`, matching edit in
`src/bullet_convert.rs` and `src/nnue.rs::output_bucket`. Nets
trained with one layout aren't inference-compatible with the other,
so this requires a full retrain.

**Path:**
- First, SB200 probe: Reckless-bucket-layout vs current-layout with
  everything else fixed. SPRT at [0, 5]. Cheap validation.
- If positive, bundle into the SB800 shelved-net run (lever 1 above):
  Reckless-KB + factor + Reckless-output-buckets combined.

Plausible gain: 5-15 Elo. Empirically untested for us; Reckless
itself uses it as standard so they've validated it in their
architecture.

### 3. Move-ordering-stat tuning as diagnostic (~? Elo, novel)

**New experimental direction** surfaced 2026-04-24. Instead of
(only) optimizing SPSA for SPRT Elo, optimize for bench-time move
ordering statistics — specifically treat them as a surrogate
objective or as a diagnostic signal.

Our bench already outputs everything needed:
- First-move cut rate (81.6% in recent bench; top engines >85%)
- Average cutoff position (1.96; ideal 1.0)
- EBF (1.67; Reckless achieves 1.55-1.60)
- Per-category cutoff rates (NMP, RFP, LMP, Futility, Hist, SEE)
- NMP verify success rate

**Two framings with different value:**

**Framing A — deployable alternative tuner.**
Run SPSA locally driven by a composite ordering-stats objective.
Risk: bench-stat optimization overfits to the 8 bench positions;
parameter sets that look beautiful on bench can tank in games.

**Framing B — diagnostic for "stuck" tunables.** The unique value.

Some SPSA tunables don't move under SPRT optimization — parameters
stay near their starting values despite many iterations. Multiple
possible reasons:
- At genuine Elo optimum ✓ (no action needed)
- In flat Elo landscape (noise-dominated) ✓ (no action needed)
- In local optimum preventing exploration (may want new gates)
- **Too-coarse parameterization**: one parameter controls behaviors
  that want different values in different contexts. SPSA finds the
  single "compromise" value. Elo is suboptimal but bounded by the
  compromise.

Running SPSA against move-ordering objectives would surface which
parameters HAVE gradient in tree-shape terms even when they have no
gradient in Elo terms. Those are the candidates for parameterization
refinement — splitting one tunable into gated variants, adding
interaction parameters, etc.

**Concrete first experiment (weekend-scope):**

No new tuner needed initially. Run bench with multiple known
parameter sets and compare stats:
1. Current main (post-retune #747)
2. Tune #750 current values
3. Tune #743 converged values (pre-merge)
4. Hand-designed "pure aggressive" and "pure conservative" reference points

Produces a table correlating parameter settings to ordering stats to
known Elo deltas. Two questions pop out:
- Does any single stat tightly correlate with Elo?
- Are there parameter sets where stats and Elo disagree significantly?

Divergences identify the stuck-tunable candidates. If nothing
diverges, we've learned SPRT is measuring everything that matters
for ordering and no new ground is uncovered.

If positive signal: investigate specific stuck tunables by adding
conditional logic. Example: if NMP_BASE_R is stuck at 5 in SPRT but
ordering-tune wants 7, split into NMP_BASE_R_PV and NMP_BASE_R_CUT,
retune. Each such split is a potential +2-5 Elo.

Plausible total contribution: 10-30 Elo over 3-5 stuck-tunable
refinements, if the diagnostic uncovers any.

### 4. NPS / cache-residency improvements (~10-25 Elo)

From the broader portfolio, specifically:
- **Group-lasso sparsity** (SB200 probe running on GPU host
  2026-04-24) — if row-level sparsity ≥40%, compacting the 49 MB
  threat matrix into L3 cache residency territory = step-function
  NPS gain across the entire fleet
- **Viridithas-style L1 permutation** — cluster hot activations for
  branch efficiency and cache locality
- **VNNI compile-time dispatch** — replace 82 runtime-flag reads
  with cfg-target-feature gates

Individually modest; collectively in the +10-25 Elo band via the
sigmoid (effective depth gain at ultra-bullet TC).

### 6. Post-improvement retune cycles (~10-20 Elo)

Every large change above shifts the search landscape. Retune-on-branch
captures flywheel multiplication. Expected pattern:

- Shelved-net SB800 deploys → 25K-iter SPSA captures +5-10 Elo
- Output-bucket change → focused NMP+RFP cluster retune captures +3-5
- Stuck-tunable refinements → whole new SPSA basin, another retune

Retunes aren't creative work but they're the cash-out mechanism for
every other lever. Don't skip them.

## Ordering principle

**Biggest leverage first, most validated first.** The shelved-net SB800
should be the next training-cycle priority. Search/NPS work proceeds
in parallel because it doesn't compete for GPU time. Correctness audits
are interleaved work for days when GPUs are busy and OB is full.
